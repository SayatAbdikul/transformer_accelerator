  Debug Analysis & Plan

  Architecture Overview (Relevant Signal Chain)

  The STORE dispatch path has this exact signal chain:

  fetch_unit (comb)        taccel_top (FF)         decode_unit (comb)
  taccel_top (comb)       dma_engine (FF)
  insn_data ─────────────► insn_data_q ──────────► insn.opcode ──────────►
  insn.opcode==OP_STORE ──► is_store port
  insn_valid ─────────────► insn_data_q WE         (always_comb)           (port
   expr)               is_store_q latch

                        state <= is_store ? AW : AR

  control_unit (FF+comb)
  state==S_ISSUE + OP_LOAD/STORE ──────► dma_dispatch
  ──────────────────────────────────────────────► dispatch port

  The critical timing for a STORE instruction:

  ┌─────────┬─────────────┬─────────────────────────────────────────────────┐
  │ Posedge │    Event    │                   Key Signals                   │
  ├─────────┼─────────────┼─────────────────────────────────────────────────┤
  │         │ fetch       │ insn_valid=1, insn_data_q <= STORE (NBA), ctrl  │
  │ K       │ delivers    │ S_FETCH→S_ISSUE (NBA)                           │
  │         │ STORE       │                                                 │
  ├─────────┼─────────────┼─────────────────────────────────────────────────┤
  │         │             │ ctrl=S_ISSUE, dma_dispatch=1 (comb),            │
  │ K+1     │ dispatch    │ is_store=insn.opcode==OP_STORE (comb from       │
  │         │             │ insn_data_q)                                    │
  └─────────┴─────────────┴─────────────────────────────────────────────────┘

  At K+1, DMA latches: is_store_q <= is_store, state <= is_store ? D_STORE_AW : 
  D_LOAD_AR.

  Known Symptoms

  1. $display in taccel_top at dispatch: insn_data_q=0x4000000002800000, 
  opcode=08, is_store=1 (CORRECT)
  2. $display in dma_engine at same $time: is_store=0 → D_LOAD_AR (WRONG)
  3. axi_write_slave never detects AW handshake (consequence of #2 — DMA never
  enters D_STORE_AW)
  4. All 8 Verilator DMA tests pass; 6/8 cocotb DMA tests fail (only OOB fault
  and async dispatch tests pass)

  Hypotheses (ranked by likelihood)

  ---
  H1: cocotb/Icarus AXI slave timing — stale r_valid causes wrong instruction 
  delivery (HIGH)

  The AXI read slave in dram_model.py has a fundamental handshake timing issue
  with Icarus:

  1. Slave sets r_valid=1 and enters inner wait loop checking r_ready
  2. At the handshake posedge, r_ready=1 in the ACTIVE region (fetch_unit in
  S_WAIT_R), but after NBA fetch_unit transitions to S_IDLE → r_ready=0
  3. If cocotb reads POST-NBA values (via cbReadWriteSynch in Icarus), the slave
   sees r_ready=0 and never breaks
  4. r_valid stays asserted. When the fetch_unit later re-enters S_WAIT_R for
  the NEXT instruction, it sees the stale r_valid=1 and accepts the OLD r_data —
   delivering the wrong instruction

  Impact: If the instruction pipeline delivers instructions from the wrong AXI
  beat, the instruction register could hold something other than STORE when
  dispatch fires. The RTL $display reading insn_data_q directly might see the
  correct value (if it fires in a specific delta), while the combinatorial
  is_store wire through decode_unit might see a different value due to
  evaluation ordering.

  Test procedure:
  - Add $display in fetch_unit at every state transition showing insn_valid,
  insn_data, state
  - Add $display showing m_axi_r_valid, m_axi_r_ready, m_axi_r_data at every
  posedge
  - Check if the same R beat is consumed twice by the fetch_unit
  - In the Python slave, print what r_ready value it reads at each inner loop
  iteration

  ---
  H2: AR request silently lost — slave busy when fetch issues new AR (HIGH)

  Related to H1. The slave initializes m_axi_ar_ready=1 and never deasserts it.
  When the slave is inside the inner R-beat wait loop, it can't process new AR
  requests. But from the DUT's perspective, ar_ready=1, so the AR handshake
  "completes" — the fetch_unit transitions to S_WAIT_R. The slave never enqueues
   this AR. The DUT fetch_unit is now stuck in S_WAIT_R waiting for an R beat
  that will never come.

  BUT: If the stale r_valid from H1 is still asserted, the fetch_unit WILL see
  it and accept the OLD r_data — getting the wrong instruction.

  Impact: After the first 16-byte block (2 instructions), every subsequent fetch
   gets data from the WRONG AXI beat. The entire instruction sequence is
  corrupted.

  Test procedure:
  - In axi_slave, deassert ar_ready when entering the inner wait loop, re-assert
   when exiting
  - Track how many AR requests the slave sees vs how many the DUT issues
  - Add a counter in the slave for enqueued vs served requests

  ---
  H3: cocotb/Icarus write slave timing — misses 1-cycle AW handshake (MEDIUM)

  Even if the DMA correctly enters D_STORE_AW, the write slave has the same
  timing issue: aw_valid is combinatorial and drops to 0 after 1 NBA cycle. The
  write slave reads POST-NBA and misses it.

  Impact: Even if H1/H2 are fixed and DMA reaches D_STORE_AW, the write slave
  won't detect the AW handshake, and STORE data won't reach DRAM.

  Test procedure: Add Timer(1, "ps") delay or ReadOnly trigger before reading
  aw_valid in write slave.

  ---
  H4: Icarus enum comparison in port expression (LOW)

  The expression insn.opcode == OP_STORE in the port connection
  .is_store(insn.opcode == OP_STORE) might not evaluate dynamically in Icarus.
  Icarus has known issues with complex expressions in port connections.

  Impact: is_store would always be 0 regardless of instruction.

  Test procedure: Replace with an explicit intermediate wire:
  logic dma_is_store;
  assign dma_is_store = (insn.opcode == OP_STORE);
  // then: .is_store(dma_is_store)

  ---
  H5: insn_data_q update races with dispatch in Icarus delta cycles (LOW)

  If insn_valid_w=1 at the same posedge as dma_dispatch=1, the NBA update to
  insn_data_q and the DMA's reading of the combinatorial is_store might
  conflict. The $display statements in different always_ff blocks may fire in
  different delta cycles, seeing different values of the combinatorial network.

  Impact: DMA latches is_store=0 despite insn_data_q having STORE bits.

  Test procedure: Add $display at the dispatch posedge showing both insn_valid_w
   and is_store in the SAME always_ff block (to eliminate delta-cycle ordering).

  ---
  Recommended Debug Order

  1. First: Run the test and capture full output to confirm exact failure mode
  2. H4 (quickest to test): Add intermediate wire for is_store in taccel_top.sv
  — 2-line change
  3. H1+H2 (most likely root cause): Fix the AXI read slave's r_ready detection
  — either use ReadOnly trigger or add a Timer(1, "ps") before reading handshake
   signals
  4. H3: Fix the AXI write slave's aw_valid detection with the same approach
  5. H5: Add combined $display diagnostics if prior fixes don't resolve

  Key Observation

  The fact that all Verilator tests pass but Icarus cocotb tests fail strongly
  suggests the RTL is correct and the issue is in the cocotb testbench 
  interaction with Icarus VPI scheduling. Verilator has a different
  (cycle-based) evaluation model that doesn't have these delta-cycle races.

  Shall I proceed with implementing fixes starting from H4 (quickest test) and
  H1+H2 (most likely root cause)?
